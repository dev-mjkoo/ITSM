('260615) ITSM 가이드

발표 순서는 오류상황판, 이벤트 조회, 이벤트 룰 관리, 문제관리 등록, 문제관리 조회 순서로 진행합니다. 화면에 구성된 그리드 컬럼과 UI 배치는 실제 구현 희망안 기준입니다.

2026.06.15 발표안

전체

여기 구성된 그리드 컬럼이나 UI 배치는 실제로 이렇게 구성되었으면 하는 형태로 미리 만들어 둔 것입니다. 그래서 별도로 "어떤 정보가 어떻게 보였으면 좋겠어요?"라고 다시 질문하지 않아도 됩니다.

  • 오늘은 오류상황판에서 실시간 오류를 확인하고, 이벤트 조회에서 전일자 취합 이벤트를 관리하고, 룰 관리와 문제관리까지 이어지는 흐름으로 설명드리겠습니다.
  • 화면은 MVP지만 컬럼 구성, 버튼 위치, 상세 팝업의 필드 구성은 실제 운영 화면에서 필요한 정보 기준으로 잡아두었습니다.

오류상황판

  • 먼저 오류상황판입니다. 상단의 형상관리, 설정, 새로고침, 일시정지 버튼은 기존에 있던 기능과 동일한 성격입니다.
  • 다만 설정 버튼을 열어보면, 조회시간, 반복주기, 파트, 채널유형, 단계, 오류구분, 거래코드 같은 조건을 이런 식으로 설정할 수 있었으면 해서 팝업 형태로 구성해 두었습니다.
  • 그리고 새로 추가된 버튼이 이벤트 조치하기입니다. 이 버튼은 조금 있다가 이벤트 조치접수 흐름에서 다시 설명드리겠습니다.
  • 이제 오류상황판의 Row를 하나 클릭해보겠습니다. Row를 클릭하면 상세 팝업이 뜨고, 탭은 상세내용과 이벤트 조치접수 두 개로 나뉩니다.
  • 상세내용 탭에는 기존에 받을 수 있는 데이터를 기준으로 오류 요약, 거래 정보, 일시 및 전문, 운영 정보처럼 비슷한 정보끼리 그룹핑해서 보여줄 수 있도록 섹션을 나누었습니다.
  • 이벤트 조치접수 탭은 오류상황판에서 해당 이벤트 담당자가 바로 조치담당자, 조치구분, 조치예정일, 조치예정내용을 입력하고 저장할 수 있게 하는 영역입니다.
  • 궁극적으로는 상황반에 "어떤 내용으로 언제까지 조치하겠습니다"라고 별도로 말하지 않아도, 오류상황판에 해당 이벤트가 나왔을 때 여기에서 바로 입력할 수 있게 하려는 목적입니다.
  • 여기서는 조치완료일을 바로 입력하는 용도가 아니라, 조치 예정 정보까지만 먼저 저장하는 용도입니다. 저장 후에야 조치완료일 입력과 결재요청 단계로 넘어갑니다.
  • 마지막으로 우측에 에러확인문구 버튼이 있습니다. 이 버튼을 누르면 상황반이나 업무팀에 바로 복사해서 요청할 수 있는 텍스트가 만들어집니다.
  • 이 포맷은 개발요건에도 적어두었기 때문에, 실제 개발 시에는 해당 양식을 기준으로 맞추면 됩니다.

이벤트 조회

  • 이벤트 관리에는 이벤트 조회와 이벤트 룰 관리 탭이 있습니다. 먼저 이벤트 조회 페이지를 보겠습니다.
  • 이 화면은 과거 이벤트를 조회하고, 실제 조치 여부까지 관리하기 위해 만든 화면입니다.
  • 오류상황판에 보인 이벤트가 여기에도 즉시 보이는 구조는 아닙니다. 하루 동안 발생한 이벤트를 모아서 다음날 중복 건 취합, 오류상황판에서 입력해둔 조치예정정보 반영 같은 정제를 거친 뒤 표시하는 구조입니다.
  • 예를 들어 6월 15일에 오류상황판에 A 이벤트가 5개, B 이벤트가 3개 발생했다면, 6월 16일 이벤트 조회에서는 A 이벤트 Row 1개에 중복 5건, B 이벤트 Row 1개에 중복 3건으로 총 2줄이 보이는 방식입니다.
  • 상단에는 조회에 필요한 검색 조건을 모두 배치해두었습니다. 오류구분, 조치구분, 발생일시, 담당부서, 담당자, 서비스명, 진행상태 기준으로 조회할 수 있습니다.
  • 우측에는 Excel 다운로드 버튼과 결재요청 버튼이 있습니다. 결재요청은 진행상태가 조치중인 이벤트들만 선택되어 있을 때만 가능합니다.
  • 이벤트 Row를 하나 클릭해보면 상세정보 탭과 최근내역 탭이 있습니다.
  • 상세정보 탭에는 상세정보 섹션과 이벤트 접수 섹션이 있습니다. 상세정보 섹션의 UI는 ITBPI와 유사한 형태로 실제 화면도 이렇게 갔으면 해서 구성해두었습니다.
  • 하단의 이벤트 접수 섹션은 조치담당자, 조치구분, 조치예정일, 조치완료일, 조치내용 필드를 가지고 있습니다. 이 UI/UX는 앞에서 본 오류상황판의 이벤트 조치접수 탭과 동일합니다.
  • 여기서도 조치담당자, 조치구분, 조치예정일, 조치내용을 먼저 입력하고 저장해야 그 다음에 조치완료일을 넣을 수 있습니다. 조치예정일과 조치완료일을 동시에 입력해서 저장하는 구조가 아닙니다.
  • 조치예정일까지 입력하고 저장하면 버튼이 결재요청으로 전환되어 보입니다.
  • 중요한 포인트는, 전일자 오류상황판의 이벤트 조치접수 탭에서 조치 관련 데이터를 입력해두었다면, 익일 이벤트 조회 시 이 이벤트 접수 섹션에 해당 값들이 채워져 있어야 한다는 점입니다.
  • 업무팀이 같은 내용을 두 번 입력하지 않게 하려는 것이 핵심입니다.
  • 두 번째 최근내역 탭은 거래코드를 기준으로 최대 100건의 오류 데이터를 보여줍니다. 물음표 도움말을 보면 기준을 확인할 수 있습니다.
  • 여기서는 에러코드가 달라도 거래코드가 같으면 같은 서비스의 과거 내역으로 보고 같이 보여주는 구조입니다.

이벤트 룰 관리

  • 이제 이벤트 룰 관리 탭으로 이동하겠습니다.
  • 여기에 이벤트 룰을 등록하면 오류상황판에서는 해당 조건에 맞는 오류를 단계가 제외인 건으로 식별할 수 있고, 익일 이벤트 데이터 취합 시에는 해당 조건을 취합 대상에서 제외하면 됩니다.
  • 룰 등록 조건은 에러코드 단독 또는 에러코드와 거래코드 조합입니다. 거래코드만 단독으로 등록하는 것은 허용하지 않습니다.
  • 그리고 룰을 등록한다고 해서 이미 이벤트 조회에 적재된 데이터가 삭제되는 것은 아닙니다. 등록 이후 전일자 데이터 취합 단계부터 신규 표시되지 않는 구조입니다.

문제관리 등록

  • 다음은 문제관리 등록입니다. 이 화면은 상황관리 Cell 인원이 문제관리 건을 처음 등록하는 화면입니다.
  • 처음 진입하면 01 기본정보 섹션만 열려 있고, 요청부서, 요청팀, 요청자, 작성부서, 작성팀, 작성자, 제목, 내용을 입력하게 되어 있습니다.
  • 요청자와 작성자는 돋보기 검색으로 선택하고, 요청부서와 요청팀, 작성부서와 작성팀은 선택된 사용자 정보 기준으로 자동 세팅되는 구조입니다.
  • 상황관리 Cell은 기본정보까지만 입력해서 등록하고, 등록이 완료되면 작성자에게 SWING 메일로 후속 작성 요청이 전달되는 흐름입니다.
  • 그래서 등록 화면에서는 02번 이후 섹션을 직접 작성하지 않고 비활성화해 두었습니다.

문제관리 조회

  • 마지막으로 문제관리 조회입니다. 등록된 문제는 이 목록에서 확인할 수 있고, 작성자 본인 건을 클릭하면 후속 작성 팝업이 열립니다.
  • 후속 작성 팝업은 문제관리 등록 화면과 동일한 화면을 재사용하지만, 이때는 01 기본정보는 잠겨 있고 02 개요부터 08 기타까지 작성자가 입력할 수 있습니다.
  • 02 개요에서는 발생시작시간, 발생종료시간, 총 발생시간, 오류내용을 입력합니다. 총 발생시간은 시작/종료 시간을 기준으로 자동 계산됩니다.
  • 03 서비스영향도, 04 문제 발생 원인, 05 조치 내용, 06 향후 대책, 07 5-why, 08 기타까지 문제 분석과 재발 방지에 필요한 정보를 작성합니다.
  • 08 기타에는 대상 구성항목(인프라)과 관련 이벤트를 연결할 수 있도록 구성했습니다. 인프라 문제인지 서비스 이벤트 문제인지에 따라 분석 대상을 같이 묶어 관리하기 위한 영역입니다.
  • 작성 중에는 임시저장을 할 수 있고, 작성이 완료되면 결재요청으로 넘어가는 흐름입니다.
